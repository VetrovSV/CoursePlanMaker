# AGENTS.md — CoursePlanMaker

#### 1. Think Before Coding
**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.


#### 2. Simplicity First
**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.
- Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

#### 3. Surgical Changes
**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.


When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.
- The test: Every changed line should trace directly to the user's request.

#### 4. Goal-Driven Execution
**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"
- For multi-step tasks, state a brief plan:
  1. [Step] → verify: [check]
  2. [Step] → verify: [check]
  3. [Step] → verify: [check]

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.



#### 5. Оставляй краткий отчёт о том, что сделал.

#### 6. Экономь токены если это не ухудшит результат
- Сначала ищи по файлам и по точным совпадениям; регулярные выражения используй, когда нужен структурный поиск.
- Не читай лишние файлы и не повторяй чтение одного и того же контента без причины.
- По возможности опирайся на уже найденные факты, а не запрашивай одно и то же несколько раз.


## Описание проекта

CoursePlanMaker — генератор рабочих программ дисциплин (РП) и фондов оценочных средств (ФОС) для российских вузов.

Проект содержит шаблоны документов в формате Markdown с YAML-метаданными, а также примеры готовых документов. На основе этих шаблонов и примеров создаются новые рабочие программы под различные дисциплины, направления подготовки и формы обучения.

---

## Типы документов

| Документ | Описание |
|---|---|
| **РП (Рабочая программа дисциплины)** | Основной документ: цели, содержание, трудоёмкость, компетенции, литература, ПО, МТО |
| **ФОС (Фонд оценочных средств)** | Приложение к РП: критерии оценивания, типовые задания, процедуры контроля |

## Архитектура шаблона РП

Подробно — см. файл `Шаблоны/рабочая_программа_дисциплины.md`. Шаблон состоит из:

1. **YAML Frontmatter** — метаданные дисциплины (код, название, часы, ЗЕТ, направление, профиль, вуз, ФИО)
2. **Плейсхолдеры `{{ TODO_... }}`** — обязательные поля для заполнения
3. **Итераторы `{% for ... %}`** — циклы для таблиц разделов, компетенций, литературы, ПО, помещений
4. **Секции-комментарии** — навигация по 8 разделам документа

## Рабочий процесс

### Создание новой РП

1. **Посмотреть БД** — выполнить `python3 db/query.py` или `sqlite3 db/disciplines.db` для поиска дисциплины и связанных параметров (часы, семестр, ЗЕТ) по нужному учебному плану
2. **Прочитать шаблон** — `Шаблоны/рабочая_программа_дисциплины.md`
3. **Определить метаданные** — дисциплина, направление, часы, ФИО
4. **Собрать TODO-данные** — для каждого плейсхолдера в шаблоне
5. **Создать файл РП** — скопировать шаблон или сгенерировать итоговый MD
6. **Валидировать** — проверить итоги (суммы часов сходятся, все поля заполнены)

### Создание ФОС

ФОС — отдельный документ, ссылающийся на РП.
Структура и пример — см. `docs/ФОС.БазыДанных.2025.md`.

## Веб-поиск

При работе над проектом может потребоваться **веб-поиск** для получения актуальной информации. Ниже перечислены сценарии, когда поиск в интернете оправдан и необходим.

### Когда использовать веб-поиск

| Сценарий | Что искать | Пример запроса |
|---|---|---|
| **ФГОС и компетенции** | Коды компетенций (ОПК, ПК, УК), индикаторы, формулировки из образовательных стандартов | «ФГОС 09.03.01 Информатика и вычислительная техника компетенции ПК» |
| **Профессиональные стандарты** | Трудовые функции, необходимые умения и знания для привязки ПК | «Профессиональный стандарт программист 06.001 трудовые функции» |
| **Учебная литература** | Актуальные учебники, ISBN, авторы, издательства, год издания | «Теория вычислительных процессов учебник 2024» |
| **ЭБС и базы данных** | Ссылки на электронно-библиотечные системы, названия и URL баз данных | «ЭБС Лань Информатика URL» |
| **Направления и профили** | Уточнение профилей подготовки, соответствие направлений | «09.03.01 профили подготовки программное обеспечение» |
| **ПО и лицензии** | Актуальные версии ПО, типы лицензий, системные требования | «VirtualBox последняя версия системные требования 2025» |
| **ГОСТы и нормативные документы** | Требования к оформлению РП, структура ФОС | «ГОСТ оформление рабочей программы дисциплины вуз» |

### Когда НЕ использовать веб-поиск

- Если информация уже есть в шаблоне или примерах проекта
- Если значение задано пользователем напрямую (метаданные, часы, ФИО)
- Для генерации художественного или творческого контента (цели, задачи — их формулирует пользователь)
- Для проверки тривиальных вычислений (сумма часов, ЗЕТ)

### Правила работы с результатами поиска

1. **Проверяй дату** — ФГОС и образовательные стандарты обновляются. Приоритет — актуальным версиям
2. **Цитируй источник** — если ссылаешься на конкретный документ, сохрани ссылку или название документа
3. **Не выдумывай** — если поиск не дал результата, сообщи об этом пользователю, а не придумывай значения
4. **Учитывай регион** — образовательные программы ориентированы на РФ, ищи информацию в российских источниках
5. **Критически оценивай** — найденные формулировки компетенций должны соответствовать актуальному ФГОС, а не быть устаревшими

## Конвенции и правила

### Наименование файлов

- **РП:** `docs/РП.{Дисциплина}.{Год}.md` — например `docs/РП.ТеорияВычислительныхПроцессов.2025.md`
- **ФОС:** `docs/ФОС.{Дисциплина}.{Год}.md` — например `docs/ФОС.ТеорияВычислительныхПроцессов.2025.md`

### Расчёт часов

При заполнении таблиц следить за консистентностью:

- `ЛК + ПЗ + ЛР = Ауд`
- `Ауд + СРС + Аттестация = Всего`
- Суммы по строкам в таблице 3.1.1 должны сходиться с итоговыми

### Работа с примерами

- `examples/примерРП.md` — это **очищенный** Markdown. Используй как референс.
- `examples/ПримерФОС.*.pdf/docx` — референс для ФОС.

### Форматирование

- Таблицы — стандартные pipe-таблицы Markdown
- Списки — дефисы или цифры
- Заголовки — `# H1`, `## H2`, `### H3` (разделы РП идут с 1 по 8)
- Жирный — `**текст**` для акцентов
- **Не использовать** HTML-таблицы, блочные `>`, символы переноса строк из PDF

### Типовые изменения относительно примера

При создании новой РП меняются как минимум:
- Название дисциплины, шифр
- Направление, профиль
- Год набора
- ФИО разработчика и декана
- Цель и задачи
- Место дисциплины (пререквизиты, семестр)
- Часы (могут отличаться)
- Содержание (темы разделов)
- Компетенции и индикаторы
- Списки литературы
- ПО
- Помещения
- Методические рекомендации (можно адаптировать из примера)

> Оптимально: 2-3 модуля, 3-5 разделов.

### Промпт для создания ФОС

> Запускается **после** того, как РП готова.
> Укажи имя файла готовой РП, чтобы я мог вытащить компетенции, темы и часы.

**Создай ФОС к дисциплине:**

- Файл РП: [docs/РП.ТеорияВычислительныхПроцессов.2026.md]
- Количество вариантов заданий: [2-3]
- Количество вопросов к зачёту/экзамену: [15-25]
- Глубина детализации: [стандартная / минимальная — чтобы не было много текста]

> **Важно:** ФОС должен строго соответствовать РП по:
> - компетенциям и индикаторам
> - названию дисциплины, коду, направлению
> - форме аттестации
> - количеству модулей/разделов (оценочные средства привязаны к темам)
>
> Оформление ФОС должно быть готово к экспорту в PDF:
> - чёткая структура разделов 1-4
> - таблицы с границами
> - единый стиль
